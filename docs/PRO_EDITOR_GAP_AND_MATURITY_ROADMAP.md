## 目标

这份文档回答两个问题：

1. 当前项目与 `Final Cut Pro / DaVinci Resolve / Premiere Pro` 这类专业剪辑软件相比，核心差距在哪里
2. 如果希望把项目做成成熟、高效、可持续演进的产品，应该按什么顺序推进

本文档不把目标定义为“立刻复制一个通用 NLE”，而是先明确当前项目更适合做强的方向，再决定哪些能力值得补，哪些能力不应过早投入。

---

## 一、当前项目的真实定位

从现有架构看，项目更接近：

- 一个面向自动化出片的长视频渲染引擎
- 一个带桌面端外壳的媒体生产工具
- 一个以 `scan -> plan -> compile -> render` 为主干的规则化视频生成系统

当前已经具备的核心基础包括：

- 四阶段引擎主链
- Python worker 与桌面端承载
- stable long-video renderer
- proxy media manifest
- visual base cache / segment cache / chunk cache
- FFmpeg 快路径与硬件编码优先
- build report、render scheduler、chunk scheduler
- packaged worker 与 CI smoke 守护

这说明项目已经具备“自动化视频工厂内核”的雏形。

它当前的优势不是“手工精剪”，而是：

- 结构化生成
- 批量处理
- 规则化剪辑
- 自动化渲染
- 针对长视频的工程优化

所以它现在的正确目标，不是先变成一个全面替代 `Final Cut Pro` 的通用编辑器，而是先成为一个成熟的“自动化长视频生产工具”。

---

## 二、与专业剪辑软件的核心差距

### 1. 差在交互模型

专业剪辑器的核心是实时非线性编辑：

- 拖拽时间线
- 波纹编辑
- 吸附与对齐
- 多轨联动
- 局部试听
- 逐帧预览
- 即时撤销/重做
- 所见即所得

当前项目的核心模型则是：

- 计划驱动
- 参数驱动
- 渲染产出驱动

也就是说，当前项目更像“生成器 + 渲染器”，而不是“成熟交互式剪辑器”。

### 2. 差在渲染后端完成度

专业剪辑器的强项不是某一条快路径，而是：

- 大多数常见场景都已有高性能后端
- 实时预览与正式导出都有稳定路径
- GPU、代理素材、优化素材、缓存、后台渲染能系统协同

当前项目虽然已经有：

- `ffmpeg_direct_chunk`
- `ffmpeg_image_chunk`
- `ffmpeg_card_chunk`
- `video_fit / video_motion_fit`
- stable chunk concat
- hardware encoding

但快路径覆盖率仍不够高，很多真实项目场景仍会回落到 `MoviePy`。

### 3. 差在媒体管理与时间线抽象

专业剪辑器不只是会导出视频，还具备完整的媒体与时间线系统：

- 媒体库管理
- optimized media / proxy media
- 多轨视频与音频
- compound clip / nested timeline
- adjustment layer
- effect stack
- subtitle track
- timeline index

当前项目虽然有 proxy 和缓存，但还没有真正成熟的：

- 多轨时间线数据模型
- 局部重算图
- clip 依赖关系管理
- 可视化编辑级工程结构

### 4. 差在稳定性与可预期性

专业剪辑器最大的隐性优势，是可预期：

- 什么时候会重新渲染
- 什么时候只会复用缓存
- 什么时候会使用代理素材
- 什么时候会退回慢路径
- 失败后如何恢复

当前项目已经在往这个方向补：

- `build_report.json`
- `render_scheduler`
- `chunk_scheduler`
- `cache_cleanup`
- worker protocol smoke

但还没有达到“任何复杂项目都可诊断、可恢复、可解释”的成熟程度。

### 5. 差在产品层完整度

专业软件还有一整层纯产品能力：

- 自动保存
- 崩溃恢复
- 项目版本管理
- 导出预设
- 插件生态
- 字体与字幕兼容性
- 色彩链路
- 音频链路
- 大项目交互流畅度
- 跨设备兼容验证

当前项目在这部分还处于早中期工程阶段。

---

## 三、哪些差距最关键，哪些不是现在的主矛盾

### 现在最关键的差距

当前阶段最关键的差距不是 UI 花样，也不是特效数量，而是：

1. 大多数常见场景还没有进入统一高性能后端
2. 长视频真实项目的导出速度仍偏慢
3. 复杂项目的渲染行为还不够稳定、可预测、可解释
4. 工程内核虽已成形，但产品层闭环还不够完整

### 现在不是主矛盾的差距

以下能力很重要，但不应过早投入为最高优先级：

- 完整多轨可视化编辑器
- 大量复杂炫技转场
- 全量复刻专业剪辑软件交互
- 先做成通用 NLE 再考虑自动化能力

如果过早投入这些，会让项目复杂度飙升，但对当前“渲染太慢、产品不够稳”的核心问题帮助不大。

---

## 四、成熟化路线：三阶段推进

## 阶段 A：先做成熟的自动化长视频生产器

这是当前最合理、性价比最高的目标。

### 阶段目标

- 10 到 30 分钟项目能够稳定出片
- 导出速度明显优于当前基线
- 预览路径和正式导出路径边界清晰
- 缓存命中与失效行为可解释
- 桌面端分发、worker、CI、缓存策略都稳定

### 核心任务

- 提高 FFmpeg 快路径覆盖率
- 提高 stable renderer 在真实项目中的命中率
- 让 `MoviePy` 逐步退到复杂场景兜底
- 提升真实项目的可观测性
- 固化 worker 打包、回退、失败恢复链路

### 成功标准

- 大部分图片/视频常见场景不再落入重型时间线合成
- 用户可以预测预览和正式导出的性能表现
- 长项目不会频繁因缓存、路径、编码器问题失控
- 一次导出失败后，能够定位失败环节并恢复继续

---

## 阶段 B：再做成熟的项目化编辑系统

只有阶段 A 稳定后，才值得投入这一层。

### 阶段目标

- 从“自动化引擎”升级为“可持续编辑的项目系统”

### 核心任务

- 引入更清晰的项目模型
- 引入多轨抽象
- 支持非破坏编辑
- 强化代理素材、优化素材、时间线状态管理
- 引入更可视化的编辑反馈

### 要求

这一步不能只是“加 UI”，必须同时补：

- 时间线数据结构
- clip 关系管理
- 更细粒度失效图
- 项目状态持久化

否则就会出现“界面更复杂，但底层仍然是一次次全局渲染”的假成熟。

---

## 阶段 C：最后才考虑向专业剪辑器形态演进

这一阶段才接近通用 NLE。

### 阶段目标

- 提供更成熟的人机协作剪辑体验

### 核心任务

- 时间线交互
- 精细修剪
- 多轨实时预览
- effect stack
- adjustment layer
- subtitle track
- undo/redo
- 自动保存 / 崩溃恢复
- 高级导出体系

### 风险

这是复杂度最高、投入最大的阶段。

如果在阶段 A、B 没做稳之前直接进入这一阶段，项目很容易变成：

- 功能界面很多
- 但渲染仍然慢
- 行为仍然不可预期
- 用户无法真正信任它

---

## 五、当前阶段最值得做的事

如果目标是“成为成熟高效产品”，当前最应该优先做的是：

### 1. 把当前引擎彻底做成混合后端

原则：

- `FFmpeg + stable renderer + hardware encoding` 做主后端
- `MoviePy` 只做复杂兜底

不要再让大量常见场景继续走 `MoviePy`。

### 2. 持续提高快路径覆盖率

重点包括：

- 纯图片 chunk
- 静态卡片段
- 轻量图片 motion
- 安全视频 fitted 路径
- 安全字幕/overlay 白名单

### 3. 把真实项目的可观测性做强

需要做到：

- 一眼能看出慢在哪里
- 一眼能看出命中了哪些快路径
- 一眼能看出为什么失效重渲

核心依赖：

- `build_report.json`
- `render_scheduler`
- `chunk_scheduler`
- cache 统计
- encoder 统计

### 4. 先把产品做稳，再做得更像专业剪辑器

当前阶段不应优先做：

- 大量复杂交互 UI
- 过多复杂转场
- 大而全的剪辑器形态

应优先做：

- 快
- 稳
- 可解释
- 可恢复
- 可分发

---

## 六、是否需要引入其他渲染引擎

### 当前结论

当前不建议立刻整体替换现有渲染引擎。

原因不是“现有引擎已经完美”，而是：

- 当前真正的问题是快路径覆盖率不够
- 现有架构已经有较强的演进基础
- 现在整体换引擎，副作用和迁移成本会很高

### 现在更适合的策略

优先把当前架构升级成：

- 高性能后端优先
- 复杂路径兜底
- 白名单渐进扩展
- 失败自动回退

### 什么时候才该认真引入新后端

只有在满足以下条件后，才值得认真评估新后端：

1. 已经完成当前低副作用提速路线
2. 真实项目仍明显无法达到目标导出速度
3. 瓶颈已经被证实不是调度和缓存问题，而是现有核心后端能力上限

那时更合理的方式也不是“全量替换”，而是：

- 只给 stable long-video renderer 增加新 backend
- 保留现有 worker / plan / cache / fallback 体系
- 白名单渐进切流

---

## 七、哪些事情现在不该过早做

以下事项容易显得“很像成熟产品”，但如果现在过早推进，收益反而不高：

### 1. 过早做完整专业时间线 UI

如果底层渲染不够快，这会放大用户挫败感。

### 2. 过早追求复杂效果全覆盖

复杂效果会显著抬高渲染后端复杂度，容易拖慢主线。

### 3. 过早追求“像 Final Cut Pro 一样”

这个目标太大，会稀释当前阶段最重要的问题。

更合理的目标是：

- 先成为成熟的自动化视频生产工具
- 再决定是否演进成更强交互型编辑器

---

## 八、建议的产品判断标准

如果项目要真正接近成熟产品，判断标准不应该只是“能导出视频”，而应该看：

### 引擎层

- 真实长视频是否稳定提速
- 快路径覆盖率是否持续提高
- 回退逻辑是否可靠

### 工程层

- 是否容易定位慢点和失败点
- worker、打包、CI 是否足够稳
- 缓存和代理策略是否可控

### 产品层

- 用户是否能理解当前状态
- 大项目是否能持续使用
- 是否具备可恢复性和可预期性

### 架构层

- 新快路径是否能低风险接入
- 后续是否有条件引入新 backend
- 是否避免了“所有复杂度都堆在一个渲染函数里”

---

## 九、最终建议

一句话总结：

当前项目离 `Final Cut Pro` 这类专业剪辑器的最大差距，不是“少几个功能按钮”，而是：

- 还不是成熟 NLE
- 还没有把大多数常见场景都收入统一高性能后端
- 还缺稳定的项目化编辑层和完整产品层

但这不代表方向错了。

当前项目最有价值的路线不是立刻模仿专业剪辑器的全部形态，而是：

1. 先把自动化长视频生产做到又快又稳
2. 让 `FFmpeg + stable + hardware` 成为主后端
3. 让 `MoviePy` 缩到复杂兜底
4. 把真实项目可观测性、可恢复性、可分发性做成熟
5. 在这个基础上，再决定是否往通用专业剪辑器形态演进

这条路更现实，也更有机会把项目真正做成成熟产品。
