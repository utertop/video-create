## 目标

这份文档用来回答三个问题：

1. `模板匹配`
2. `AI 配乐蓝图`
3. `时间线微调`

这三项现在能不能做。

答案是：都能做，但不应并行重投入。当前更合理的顺序是：

1. 先做 `模板匹配`
2. 再做 `AI 配乐蓝图`
3. 最后做 `轻量时间线微调`

原因很明确：

- `模板匹配` 最贴近当前 `scan -> plan -> compile` 主链，落地最快，产品收益最直接
- `AI 配乐蓝图` 能明显提升成片质量，但应先做“推荐与规划”，不应一开始就走复杂自动作曲
- `时间线微调` 有必要，但当前阶段应先做结构化参数微调，而不是过早做重型专业时间线编辑器

---

## 一、当前阶段的主判断

当前项目更适合先做成：

- 一个成熟的自动化长视频生产工具
- 一个带轻量人工干预能力的结构化编辑系统

而不是立刻做成：

- 完整专业 NLE
- 高自由度实时多轨时间线编辑器

所以这三项能力的正确落法应该是：

- `模板匹配`：做成计划层能力
- `AI 配乐蓝图`：做成规划层能力
- `时间线微调`：做成结构化控制层能力

---

## 二、建议的实施顺序

### Phase 1：模板匹配

目标：

- 让系统根据素材特征自动选择更合适的剪辑模板和章节结构

最小交付：

- 输入一组素材后，系统能给出 1 到 3 个推荐模板
- 每个模板包含：
  - 节奏风格
  - 推荐分段方式
  - 开头/主体/结尾结构建议
  - 适合的转场与标题风格

### Phase 2：AI 配乐蓝图

目标：

- 不直接生成音乐，而是先输出可靠的配乐规划

最小交付：

- 针对一个项目输出：
  - BGM 风格建议
  - 分段情绪建议
  - 音乐进入点和退出点
  - 节点起伏建议
  - ducking / fade / 音量层级建议

### Phase 3：轻量时间线微调

目标：

- 允许用户在不进入重型时间线编辑器的前提下，做高价值微调

最小交付：

- 调整片段顺序
- 调整单段时长
- 调整转场类型
- 调整标题显示区间
- 调整 BGM 入点、出点、音量节点

---

## 三、模块拆分与实施清单

## 1. 模板匹配

### 为什么先做它

这项能力最符合当前架构，因为项目已经有：

- `scan`：素材分析
- `plan`：结构规划
- `compile`：渲染计划生成

模板匹配本质上是在 `scan` 和 `plan` 之间增加一个“选择器”，而不是重做渲染引擎。

### MVP 范围

第一版只做模板推荐，不做模板市场，不做复杂可视化拖拽。

建议第一批模板维度：

- 旅行
- 探店
- 日常 vlog
- 产品展示
- 图文解说

建议第一批匹配依据：

- 素材类型占比：图片 / 视频 / 竖屏 / 横屏
- 素材时长分布
- 镜头节奏密度
- 是否存在章节性内容
- 标题卡、章节卡的使用倾向

### 需要做的事

- [ ] 为 `scan` 结果补充模板匹配所需的摘要字段
- [ ] 设计模板元数据结构
- [ ] 增加模板评分器
- [ ] 在 `plan` 阶段接入模板候选
- [ ] 支持返回 Top-N 模板建议
- [ ] 支持“强制指定模板”和“自动推荐模板”两种模式

### 建议改动点

- `video_engine_v5.py`
  - `scan` 输出增加摘要字段
  - `plan` 增加模板推荐与模板落选原因
- worker 协议
  - 新增模板推荐结果透传
- 前端
  - 新增模板推荐卡片与手动切换入口

### 验收标准

- 相同素材输入能稳定输出相同模板推荐
- 模板建议结果可解释，不是黑盒随机
- 用户可手动覆盖模板建议
- 切换模板后能稳定重新生成 `render_plan`

### 暂时不要做

- 不先做用户自定义模板编辑器
- 不先做模板商城
- 不先做复杂的实时模板预览器

---

## 2. AI 配乐蓝图

### 为什么第二阶段做

这项能力对成片质量有帮助，但它不是当前最大瓶颈，也不需要先碰重型音频引擎。

正确方式不是“先自动生成整首音乐”，而是先做：

- 风格建议
- 节点建议
- 段落情绪建议

也就是让它先成为“音乐导演层”，而不是“作曲引擎层”。

### MVP 范围

第一版只输出结构化配乐蓝图。

蓝图字段建议：

- `bgm_style`
- `energy_curve`
- `intro_in`
- `chapter_lift_points`
- `drop_points`
- `outro_fade`
- `ducking_strategy`

### 需要做的事

- [ ] 从 `plan` 中提取章节、高潮点、情绪段落
- [ ] 定义配乐蓝图 JSON 结构
- [ ] 增加风格推荐器
- [ ] 增加时间段级配乐建议器
- [ ] 支持将蓝图映射到现有音频混流参数
- [ ] 在 UI 中增加“采用建议”入口

### 建议改动点

- `video_engine_v5.py`
  - `plan` 阶段增加音频蓝图生成
  - `compile` 阶段把蓝图映射为可执行参数
- 音频策略文档和配置
  - 与现有 `AUDIO_MUSIC_STRATEGY.md`
  - `AUDIO_STABLE_MIXING_STRATEGY.md` 对齐
- 前端
  - 展示每段音乐建议与总体风格建议

### 验收标准

- 蓝图能稳定生成
- 蓝图可以被现有混音参数消费
- 用户可以只接受部分建议
- 不影响现有手动 BGM 配置能力

### 暂时不要做

- 不先做自动作曲
- 不先做复杂 AI beat 对齐编辑器
- 不先引入高风险第三方在线音乐生成闭环

---

## 3. 轻量时间线微调

### 为什么第三阶段再做

这项能力是有价值的，但最容易失控。

如果一开始就把它做成：

- 多轨
- 任意拖拽
- 实时预览
- 精细帧级编辑

那很快就会把项目带到“做一个专业剪辑器”的方向，和当前阶段的核心目标冲突。

### 正确的第一版定位

只做“结构化时间线微调”，不做完整 NLE。

建议支持的微调动作：

- 调整片段顺序
- 调整单段时长
- 调整转场类型
- 调整标题显示区间
- 调整章节分隔点
- 调整 BGM 入点、出点、音量节点

### 需要做的事

- [ ] 抽象一份可编辑的结构化时间线模型
- [ ] 为关键字段提供 patch/update 协议
- [ ] 增加“微调后局部重编译”能力
- [ ] 确保现有缓存与局部失效策略可复用
- [ ] 为前端提供稳定的微调命令接口

### 建议改动点

- `video_engine_v5.py`
  - `plan` 与 `render_plan` 之间增加可编辑层
  - 支持 patch 后局部重编译
- `src-tauri/src/lib.rs`
  - 增加微调命令桥接
- 前端
  - 先做表单化和列表化编辑
  - 不先做重型可视化拖拽时间线

### 验收标准

- 微调一个片段时，不需要整项目完全重来
- 微调动作可回放、可撤销
- 微调后导出结果可预期
- 局部修改能复用既有缓存

### 暂时不要做

- 不先做复杂多轨拖拽时间线
- 不先做逐帧实时预览器
- 不先追求 Final Cut Pro 级交互

---

## 四、三项工作的共同前提

在推进这三项之前，需要坚持几个共同原则：

1. 都必须服务于当前“自动化长视频生产器”主线
2. 都应优先复用现有 `scan -> plan -> compile -> render` 架构
3. 都应优先走结构化 JSON 协议，不先引入复杂状态机
4. 都必须保留手动覆盖能力
5. 都不应破坏现有 worker、缓存、局部失效和稳定渲染路径

---

## 五、推荐的开发排期

### 第一阶段

- [ ] 模板元数据结构
- [ ] 素材摘要字段扩充
- [ ] 模板评分器
- [ ] 模板推荐 UI

### 第二阶段

- [ ] 配乐蓝图数据结构
- [ ] 段落情绪和节点提取
- [ ] 蓝图到混音参数映射
- [ ] 配乐建议 UI

### 第三阶段

- [ ] 结构化时间线模型
- [ ] 微调 patch 协议
- [ ] 局部重编译支持
- [ ] 轻量编辑 UI

---

## 六、最终建议

如果现在就要开工，最正确的顺序是：

1. 先做 `模板匹配`
2. 再做 `AI 配乐蓝图`
3. 最后做 `轻量时间线微调`

其中：

- `模板匹配` 是当前最值得先做的一步
- `AI 配乐蓝图` 是最适合作为第二层质量增强的一步
- `时间线微调` 必须做，但第一版一定要克制，避免把项目过早带向重型专业编辑器路线

一句话总结：

先增强“系统自动给出更好的方案”，再增强“用户对方案做轻量微调”，而不是先去做一个完整时间线编辑器。
