# Video Create Studio V5 渲染引擎差距与升级路线图

## 1. 文档目标

这份文档回答四个问题：

- 当前项目现有的三种渲染方式，和 Final Cut Pro / DaVinci Resolve / Premiere Pro 这类成熟剪辑软件相比，差距主要在哪里。
- 为什么当前项目在 20 分钟左右、同时包含较多照片和视频的项目里，正式生成耗时会明显偏长。
- 下一步最值得投入的优化项是什么，哪些会真正改善长视频效率，哪些只是“看起来多了一个模式”。
- 如何在不推翻现有 UI 的前提下，把渲染系统逐步升级为更成熟的工程架构。

这份文档不追求“理论上最强”，而是追求：

- 可在当前项目结构里落地
- 能和现有 `FFmpeg 快速拼接 / MoviePy 交叉淡化 / 自动选择` 共存
- 能优先改善长视频、照片较多项目的真实体验

## 2. 当前引擎现状

当前项目表层上有三种渲染方式：

- 自动选择
- FFmpeg 快速拼接
- MoviePy 交叉淡化
  
从工程角度看，它们本质上更接近：

- 一条偏快速、偏稳定的执行路径
- 一条偏视觉效果、偏灵活的执行路径
- 一条由系统做初步分流的默认路径

这套设计对早期版本是合理的，因为它能快速把“直切”和“轻转场”两类核心需求先跑起来。

但它和成熟剪辑软件的差距也正来自这里：

- 这些更像“导出方式”
- 还不是“完整时间线渲染架构”

成熟软件真正强的地方，不在于下拉菜单里有几个选项，而在于内部会对素材、时间线、缓存、代理、编码、局部失效做大量分层调度。

## 3. 和成熟软件的核心差距

### 3.1 当前更像一次性导出，成熟软件更像时间线引擎

当前项目的主思路仍然偏向：

- 生成 render plan
- 按段渲染
- 最终导出整片

成熟软件的思路更偏向：

- 时间线节点化
- 局部缓存
- 局部失效
- 局部复用
- 最终只导出变化部分

这意味着成熟软件在以下场景里优势很大：

- 改了一个标题，不需要整条视频都重新重渲
- 改了音乐，不需要画面重算
- 改了单个章节转场，不需要前后所有内容都重新处理
- 预览和正式导出使用不同素材层级

而当前项目在不少情况下还是偏“整段或整 chunk 重新生成”。

### 3.2 当前对图片段和视频段的处理还不够分层

20 分钟左右的视频里，如果混合了大量照片，性能瓶颈经常并不在“视频段本身”，而在照片段反复做这些工作：

- 原图读取
- EXIF 修正
- 缩放
- 补背景
- 模糊
- 轻微镜头运动
- 转为最终帧流

成熟软件通常会把照片素材单独纳入一套更强的优化链路：

- 先生成 optimized media
- 先生成代理图像
- 把静态动画参数化
- 把高频复用的照片段提前烘焙成缓存片段

而不是每次正式导出时再从原图重新做整套处理。

### 3.3 当前底层“引擎选项”还不等于“渲染调度器”

成熟软件内部不是简单地选择：

- FFmpeg
- 某个 Python 库
- 某个转场路径

而是先判断：

- 这个 segment 是否能纯滤镜表达
- 这个 chunk 是否可以 copy concat
- 哪些段必须进入更重的时间线系统
- 哪些效果可以预先烘焙
- 哪些素材应该走代理
- 哪些资源应该在后台预处理

也就是说，成熟方案的关键不在“多一个引擎”，而在“更聪明的调度层”。

### 3.4 当前对长视频的优化已经有基础，但还没形成完整闭环

项目已经具备了不错的基础：

- 稳定模式
- chunk 渲染
- render cache
- audio cache
- FFmpeg concat 优先
- Python worker 方向

这些已经把项目从“纯实验原型”推进到了“可实用工程”。

但和成熟软件相比，还缺三层闭环：

- 照片段专项缓存闭环
- 预览代理闭环
- 局部失效闭环

## 4. 为什么 20 分钟视频会明显慢

当前项目在 20 分钟视频里耗时明显偏长，通常不是单一原因，而是几类开销叠加：

### 4.1 图片段重复处理过多

当项目里存在较多照片时，图片段如果每次都重新执行：

- 读图
- 缩放
- 背景模糊
- 轻运动
- 帧合成

最终就会比用户直觉里“图片应该很轻”慢很多。

### 4.2 MoviePy 仍在承担部分重时间线工作

虽然项目已经在扩大 FFmpeg 覆盖率，但只要某些段仍然需要：

- 多层 Composite
- 图片动效逐帧生成
- 标题叠加
- 局部转场

MoviePy 这条路径在长视频里仍然会很吃 CPU、内存和总时长。

### 4.3 预处理和正式导出之间的复用还不够彻底

如果同一段照片动画、章节卡、文字卡、轻镜头运动没有被充分复用，就会出现：

- 逻辑上看起来没改
- 实际上又算了一遍

这是很多“越长越慢”的真实来源。

### 4.4 长视频里最终收尾导出仍然很重

即使 chunk 渲染已经分段完成，最后一步仍可能包含：

- 最终视频拼接
- 最终音频混音
- 最终编码输出

如果前面没有把更多结果变成“可直接复用的中间产物”，最后这一步仍会很重。

## 5. 当前不建议做的方向

为了避免投入很多时间却没有核心收益，这几类方向不建议作为下一步优先项：

### 5.1 先增加第四个“底层渲染方式”选项

比如再加一个新的下拉选项，看起来像功能增强，但如果背后还是同样的时间线模式和缓存粒度，收益有限。

问题核心不是“渲染方式太少”，而是“调度、缓存和复用不够成熟”。

### 5.2 先堆更多花哨效果

如果底层照片段缓存、局部失效、代理链路还不够成熟，再继续堆更复杂的：

- 图片特效
- 多层字幕动效
- 更重的镜头运动

只会继续放大长视频成本。

### 5.3 先做复杂时间线 UI

时间线当然是长期重要能力，但在渲染架构没进一步打稳前，先大做时间线 UI，往往会把开发精力从更关键的底层性能优化上分走。

## 6. 推荐的成熟升级路线

下一步建议不是“再加一个引擎”，而是把现有三条路径升级成一套成熟的分层架构。

### P1. 照片段专项优化

这是当前最值得优先投入的一项。

目标：

- 让“长视频 + 很多照片”的项目明显提速
- 减少照片段重复处理

建议实现：

- 原图修正结果缓存
- 背景模糊结果缓存
- 目标分辨率缩放缓存
- 照片动画段缓存为短视频片段
- 对相同参数的照片段直接复用缓存片段

建议原则：

- 不再把照片视为“每次导出都现算”的轻素材
- 而是把它视为“最适合提前烘焙”的素材类型

为什么优先做：

- 你的长视频场景里明确包含较多照片
- 这一项对导出速度改善往往最直接
- 也最容易和当前架构衔接

### P2. 扩大 FFmpeg 纯路径覆盖率

目标：

- 让更多 segment 不进入 MoviePy
- 减少 Python 持有重时间线的范围

建议优先覆盖：

- 直切
- 轻淡化
- 单图轻运动
- 轻量章节卡
- 单层文字叠加
- 轻量 BGM 混音

策略：

- 能被滤镜图表达的，尽量不要交给 MoviePy
- 把 MoviePy 留给“确实无法简单滤镜表达”的复杂段

这一步的意义是：

- 不是新增模式
- 而是让现有模式更大比例走快速路径

### P3. 做真正的局部失效和段级复用

目标：

- 改一处，不重渲全部

建议实现：

- segment cache key 更稳定
- 图片段缓存
- 章节卡缓存
- 标题卡缓存
- 改音乐时不重渲画面
- 改文字时不重渲整片
- 改单段转场时只失效局部

这是你向成熟剪辑软件靠近非常关键的一步。

### P4. 代理素材和真实低清预览体系

目标：

- 让预览和正式导出彻底分层

建议实现：

- 扫描阶段生成低清代理
- 低清预览一律走代理
- 正式导出走原素材
- 超大图片先生成 optimized image
- UI 里让用户知道当前看到的是“预览代理”还是“正式输出”

这会直接改善两个体感：

- 调参更快
- 试错成本更低

### P5. 引入真正的渲染调度器

目标：

- 把当前“引擎选择”升级为“自动执行规划”

调度器职责应当包括：

- 判断 segment 是否可纯 FFmpeg
- 判断 chunk 是否可 concat copy
- 判断哪些段先预烘焙
- 判断是否强制代理
- 判断是否启用稳定模式
- 判断哪些步骤可后台并行

这是中长期最有价值的方向。

## 7. 对标成熟软件时应学习的不是 UI，而是这些机制

下一步真正值得参考主流剪辑软件的，不是“界面像不像”，而是下面这些机制：

### 7.1 Optimized Media

素材先转为更适合编辑和渲染的中间格式，减少后续重复解码成本。

### 7.2 Proxy Media

预览不拿最终素材硬扛，而是低清代理先跑通创作链路。

### 7.3 Render Cache

复杂段先缓存，不改就不重算。

### 7.4 Background Render

用户在调参数或停留时，系统后台悄悄预处理后续可能会用到的段。

### 7.5 Smart Timeline Invalidation

不是任何改动都让整条时间线失效，而是只失效必要片段。

## 8. 对当前项目的落地建议

结合当前代码基础，推荐你这样推进：

### 第一阶段：先补照片段缓存体系

这一步最容易在现有架构里得到真实提速。

优先产出：

- 图片段缓存规则文档
- 图片段缓存 key 设计
- 图片段预烘焙实现
- 对应冒烟测试

### 第二阶段：扩大 FFmpeg 适用范围

重点不是加新 UI 选项，而是让“自动选择”和“稳定模式”更聪明。

优先产出：

- segment 分类规则
- 哪些段走 FFmpeg
- 哪些段走 MoviePy
- 哪些段必须预烘焙

### 第三阶段：把局部失效做细

这一步会开始真正改善中大型项目的日常工作流体验。

优先产出：

- 局部失效规则
- 修改不同参数时的失效边界定义
- chunk / segment / audio 三层失效设计

### 第四阶段：代理和预览闭环

这一步更偏产品体验升级。

优先产出：

- 低清代理规范
- 预览流程规范
- UI 标识和用户提示

## 9. 对当前 UI 的建议

当前 UI 不需要立刻大改。

建议原则：

- 前台继续保留创作者能理解的选项
- 后台不断增强真实调度能力

具体建议：

- “底层渲染方式”可以继续保留，但逐步弱化为高级项
- 主流程更多强调：
  - 剪辑策略
  - 性能档位
  - 预览方式
  - 长视频稳定策略

也就是说：

- 用户看到的是创作语言
- 系统内部执行的是工程语言

## 10. 推荐优先级结论

如果只能选一个下一步，建议优先做：

### 第一优先级

照片段专项缓存和预烘焙

### 第二优先级

扩大 FFmpeg 纯路径覆盖率

### 第三优先级

局部失效和段级复用

### 第四优先级

代理素材和真实低清预览

### 第五优先级

渲染调度器和后台预处理

## 11. 最终结论

当前项目和成熟剪辑软件的差距，不主要在于：

- 下拉菜单里渲染方式不够多

而主要在于：

- 图片段处理还不够专项化
- 局部缓存和局部失效不够细
- 代理和正式导出还没有彻底分层
- 调度器还不够成熟

所以真正正确的下一步不是“再加一个渲染引擎选项”，而是：

- 让现有引擎路径更聪明
- 让照片段、音频段、视频段分层优化
- 让缓存和失效粒度更细
- 让预览和正式导出彻底分层

这样项目才会从“可用的视频生成器”，逐步升级成“更接近成熟剪辑软件的渲染系统”。
