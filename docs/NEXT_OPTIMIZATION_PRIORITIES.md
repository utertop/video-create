# 下一阶段优化优先级

## 目标

当前阶段最值得投入的方向，不是继续堆叠新功能，而是持续压缩“修改后的等待时间”和“重复导出的成本”，把项目从“能跑通”进一步推进到“适合高频创作使用”。

---

## P0：局部失效 + 段级缓存

### 目标

让小改动不再触发整片重渲。

### 优先实现

- 图片段缓存
- 标题卡缓存
- 章节卡缓存
- 音频参数变更时不重渲画面
- 单段改动只失效局部 segment / chunk

### 预期收益

- 修改标题文字时，不再重渲无关片段
- 修改 BGM、音量或混音参数时，只更新音频相关结果
- 修改单段转场时，只失效必要的相邻片段
- 相同输入重复导出时，缓存命中率显著提升

### 验收标准

- 改标题文字，不重渲无关视频段
- 改音频参数，不重渲画面
- 改单段转场，只失效局部片段
- 重复导出耗时显著低于首次完整渲染

---

## P1：低清预览闭环加速

### 目标

把预览流程做成真正可高频试错的工作流，而不是“缩小版正式渲染”。

### 优先实现

- 扫描阶段生成 proxy media
- 预览流程固定走代理素材
- 正式导出固定走原始素材
- UI 明确标识“当前为预览代理”或“当前为正式输出”

### 预期收益

- 调参更快
- 试错成本更低
- 预览与正式输出职责更清晰
- 用户更容易理解当前看到的是哪一种结果

### 验收标准

- 预览耗时显著低于正式导出
- 预览与正式输出在结构、节奏、时长上保持一致
- 用户可明确区分代理预览与正式输出

---

## P2：Packaged Worker 瘦身与稳定化

### 目标

让桌面端可执行分发更轻、更稳、更可重复构建。

### 优先实现

- 精简 PyInstaller 收集范围
- 固定更稳定的 Python 依赖版本
- 减少不必要 metadata 打包
- 减少不必要测试模块或开发辅助模块进入产物
- 降低 `video-create-worker.exe` 体积

### 预期收益

- 打包体积下降
- CI 构建更稳定
- 本地与 CI 产物更一致
- 后续维护成本更低

### 验收标准

- `video-create-worker.exe` 体积下降
- bundled worker smoke test 持续通过
- CI 打包流程稳定可复现

---

## 建议推进顺序

1. 先完成“局部失效 + 段级缓存”
2. 再推进“低清预览闭环加速”
3. 最后收紧“Packaged Worker 瘦身与稳定化”

---

## 结论

如果只优先落地一个方向，当前应优先推进 **局部失效 + 段级缓存**。

原因很直接：

- 对真实使用体验提升最大
- 最能减少“改一点也要等很久”的挫败感
- 能直接提升后续预览优化与正式导出的整体效率
