# Video Create Studio V5 视频生成性能与稳定性优化方案

## 1. 问题判断

当前视频生成慢、长视频内存持续上涨、最终文件可能损坏，核心原因不是单点硬件不足，而是渲染架构进入了 MoviePy 的高内存区间：

- 标准 MoviePy 时间线会把多个 `ImageClip / VideoFileClip / CompositeVideoClip` 组合挂在同一个最终 timeline 上。
- 图片补背景、标题叠加、动效、转场都会生成额外 CompositeClip。
- 30 分钟以上或几百上千段素材时，Python 进程容易长时间持有 clip、mask、音频 reader、临时帧对象。
- 如果最后统一写出失败，前面已经渲染的时间全部浪费，输出文件也可能是半成品或损坏文件。

所以性能优化方向应当是：不要让一个 Python/MoviePy 进程一次性吃完整部长片，而是把工程拆成可恢复、可缓存、可验证的分段渲染流水线。

## 2. 产品级目标

### 2.1 普通机器不被打爆

默认策略必须适配 8GB 到 16GB 内存机器，而不是只为高配工作站设计。

- 低配机器：优先稳定，减少复杂转场，chunk 更短。
- 中配机器：保持正常画质，使用中等 chunk。
- 高配机器：允许更长 chunk、更高质量、更多并行预处理。

### 2.2 长视频必须可恢复

30 分钟以上项目不应当“一次失败全部重来”。

要求：

- 每个 chunk 单独输出 mp4。
- chunk 输出后立即校验。
- 成功 chunk 写入 manifest。
- 再次生成时复用未变化 chunk。
- 最终合成失败不覆盖旧成片。

### 2.3 预览和正式生成分离

创作者在试效果时，不应该每次都跑完整高质量渲染。

要求：

- 预览使用低清、短片段、低 fps。
- 正式生成使用完整素材和目标质量。
- UI 中所有“预览”都不传视频流，只传路径和参数。

### 2.4 创作选择权与安全执行层分离

性能优化会影响“复杂程度”和“执行方式”，但不应该取消用户的创作选择权。

这里需要把 UI 和引擎逻辑分成两层：

- 创作层：用户选择剪辑策略、章节文字风格、章节文字动效、章节背景、画幅、质量。
- 执行层：系统根据素材量、预计时长、机器压力和质量档位，决定是否启用稳定渲染、chunk 长度、转场复杂度上限、预览分辨率。

原则：

- 用户选择 `旅拍柔和`，系统不能偷偷改成 `快速成片`。
- 用户选择章节文字动效，系统应该尽量保留。
- 如果项目过大，系统可以把复杂执行降级为更安全的版本，例如减少闪切、缩短交叉淡化、降低图片运动强度。
- 降级必须可见，不能静默发生。
- 用户仍然可以强制高质量完整效果，但 UI 必须显示风险提示。

推荐提示文案：

> 检测到素材量较大，已自动启用稳定渲染。章节文字动效会保留，但部分复杂转场和镜头运动会自动降级，以避免内存过高或生成失败。

这意味着性能优化不是替代动效系统，而是给动效系统增加一个“安全执行层”。

## 3. 当前项目已有基础

项目已经具备一部分长视频稳定基础：

- `V56StableRenderer`：已支持 chunk 分段渲染。
- `chunk_manifest.json`：已记录 chunk 缓存状态。
- `build_report.json`：已记录失败 chunk 和输出报告。
- `FFmpeg concat copy`：chunk 最终拼接优先走无重编码快速拼接。
- `atomic replace`：最终视频校验通过后才覆盖目标文件。

本轮已先做两项低风险加固：

- 长片策略默认 chunk 从 `240s` 降为 `120s`，降低单 chunk 峰值内存。
- chunk cache key 纳入 `transition_config / motion_config / rhythm_config / keep_audio / python_quality`，避免策略变化后误复用旧 chunk。

## 4. 推荐渲染架构

### 4.1 标准模式

适用：

- 3 分钟以内短视频。
- 素材段数量较少。
- 需要较丰富的转场和动效。

策略：

- 可以使用 MoviePy 标准 timeline。
- 可启用 `travel_soft / beat_cut / documentary` 等创作策略。
- 失败成本低，可以整体重跑。

### 4.2 稳定模式

适用：

- 10 分钟以上视频。
- 超过 80 个 segment。
- 大量图片、视频混合素材。
- 用户选择“长片稳定”。

策略：

- 强制走 `V56StableRenderer`。
- 默认 chunk：`120s`。
- 低配机器建议：`60s` 到 `90s`。
- 中高配机器建议：`120s` 到 `180s`。
- chunk 内允许轻量转场。
- chunk 间使用 FFmpeg concat copy。

### 4.3 极速模式

适用：

- 批量审片。
- 几千张图片先快速看结构。
- 不追求复杂动效。

策略：

- 降低 fps 到 24 或 20。
- 关闭复杂镜头运动。
- 尽量使用 FFmpeg 直接生成图片序列或 concat。
- 图片默认固定展示或极轻微缩放。

## 5. 代码优化路线

### P0：立即稳定性保护

目标：减少损坏文件和内存爆炸。

- 默认长视频 chunk 降到 `120s`。
- chunk cache key 纳入剪辑策略配置。
- 最终文件必须先写 `.rendering.tmp.mp4`，校验通过后再替换。
- chunk 校验失败立即停止，不继续拼接。
- 每个 chunk 完成后 `close_clip + gc.collect`。

状态：部分已完成。

### P1：硬性内存保护

目标：防止用户机器被打爆。

建议实现：

- 渲染前估算复杂度：
  - `total_duration`
  - `segment_count`
  - `video_count`
  - `image_count`
  - `quality`
  - `transition_complexity`
- 根据复杂度自动选择模式：
  - 小项目：标准模式。
  - 中项目：自动稳定模式。
  - 大项目：强制稳定模式。
- 添加用户可见提示：
  - “检测到素材量较大，已自动启用稳定渲染，速度会略慢但更可靠。”

### P2：分段预处理缓存

目标：相同素材和参数不重复处理。

建议实现：

- 图片 EXIF 修正、缩放、背景模糊预先写入磁盘缓存。
- 视频首帧/中帧/尾帧提取写入磁盘缓存。
- 标题卡背景图写入缓存。
- 预处理缓存 key 包含：
  - 文件路径
  - 文件大小
  - mtime
  - 输出分辨率
  - 背景模式
  - 引擎版本

收益：

- 大量图片项目会明显减少重复 PIL 处理。
- 失败重跑时速度更稳定。

### P3：渲染质量自适应

目标：同一套软件适配不同机器。

建议在“底层渲染方式”和“质量”附近增加三档性能配置，让用户明确知道当前是效果优先还是稳定优先。

#### 省内存

- chunk：`60s`
- fps：`24`
- preset：`veryfast`
- 图片动效：低
- 转场：只保留直切/轻淡化

#### 平衡

- chunk：`120s`
- fps：`30`
- preset：`veryfast` 或 `medium`
- 图片动效：中
- 转场：完整策略但限制重叠时长

#### 高质量

- chunk：`180s`
- fps：`30`
- preset：`medium`
- 图片动效：完整
- 转场：完整

UI 上可以叫：

- 稳定优先
- 平衡推荐
- 画质优先

UI 关系建议：

- `剪辑策略`：表达创作意图，例如旅拍柔和、节奏卡点、纪录叙事。
- `底层渲染方式`：表达技术执行方式，例如自动选择、FFmpeg 快速拼接、MoviePy 交叉淡化。
- `质量`：表达画质目标，例如草稿、标准、高质量。
- `性能档位`：表达安全执行策略，例如稳定优先、平衡推荐、画质优先。

四者不应该互相替代，而应该互相约束：

- 用户选择 `高质量 + MoviePy 交叉淡化 + 旅拍柔和`，如果是短视频或素材较少，允许完整效果。
- 如果检测到 30 分钟以上、几百到几千个 segment、或大量高清视频，系统自动建议 `稳定优先` 或 `平衡推荐`。
- 在稳定优先下，章节文字动效保留，章节之间仍可保留轻淡化或桥接过渡，但复杂闪切、长交叉淡化、强缩放运动会降级。
- 用户可以强制 `画质优先`，但需要显示风险提示，例如“可能占用较多内存，长视频存在生成失败风险”。

推荐 UI 放置位置：

- 放在当前“底层渲染方式”和“质量”附近。
- 显示一条自动判断结果，例如“当前建议：平衡推荐，预计使用 120 秒分段渲染”。
- 当触发稳定保护时，显示黄色提示条。
- 当用户强制高质量完整效果时，显示确认提示。

推荐自动规则：

```text
短视频 / 少量素材：
  默认：画质优先或平衡推荐
  效果：完整剪辑策略、完整章节文字动效、完整轻量转场

中等项目：
  默认：平衡推荐
  效果：保留章节文字动效，限制转场重叠时长，chunk 约 120s

长视频 / 大量素材：
  默认：稳定优先
  效果：章节文字动效保留，复杂转场降级，chunk 约 60s 到 120s
```

### P4：真实低清预览

目标：减少正式生成试错。

建议：

- 对每种剪辑策略生成 2 到 5 秒低清预览。
- 分辨率：`480p` 或 `540p`。
- fps：`12` 到 `15`。
- 只取前 2 到 3 个代表 segment。
- 输出到 app cache。
- 前端只播放本地 mp4 路径。

收益：

- 用户不用完整生成才知道风格是否合适。
- 大幅减少无效正式渲染。

### P5：FFmpeg 优先路径

目标：能不用 MoviePy 的地方尽量不用 MoviePy。

适合 FFmpeg 的任务：

- 纯直切拼接。
- chunk 最终拼接。
- 批量图片转视频。
- 简单淡入淡出。
- 音频拷贝或简单混音。

仍建议保留 MoviePy 的任务：

- 复杂标题卡。
- 章节背景桥接。
- Python/PIL 文本渲染。
- 个性化动效组合。

长期方向：

- MoviePy 负责“生成复杂片段”。
- FFmpeg 负责“批量编码和最终拼接”。

## 6. Python 打包分发方案

你的判断是对的：成熟桌面产品不能要求用户安装 Python。

推荐路径：

### 阶段 1：PyInstaller 打包 CLI

先把 `video_engine_v5.py` 打成独立 exe。

优点：

- 改动最小。
- Tauri 当前命令行调用方式可以继续使用。
- 先解决用户环境依赖问题。

风险：

- MoviePy、imageio-ffmpeg、Pillow 体积较大。
- 打包产物需要做依赖探测和路径修正。

### 阶段 2：本地 HTTP Worker

再升级为本地 Python worker 服务。

推荐：

- FastAPI 或更轻量的 `http.server + queue`。
- Tauri 启动时拉起本地 worker。
- 前端/Tauri 只发任务 JSON。
- worker 返回 job_id。
- 进度通过 stdout JSON、HTTP polling 或本地 websocket。

收益：

- 不需要每次任务重新启动 Python。
- 可以管理任务队列。
- 可以暂停、取消、恢复。
- 可以统一管理缓存、临时目录、资源上限。

注意：

- 本地服务只监听 `127.0.0.1`。
- 必须带随机 token，避免本机其他程序误调用。

## 7. 前端 Timeline 性能方案

你的判断也是对的：不要从零手写大型视频时间线。

推荐原则：

- 前端 timeline 只做“状态编辑”。
- 不在 WebView 里真实渲染视频。
- 不把视频帧大量搬进前端内存。
- 缩略图走磁盘缓存路径。

后续如果做复杂 timeline：

- 优先找 Canvas/WebGL 时间线库。
- segment 数据结构保持轻量：
  - `segment_id`
  - `source_path`
  - `start`
  - `duration`
  - `thumbnail_path`
  - `transition_config`
  - `motion_config`
- 前端只渲染缩略图和轨道状态。

## 8. IPC 与大文件处理规则

这条必须作为项目硬规则：

- IPC 只传 JSON 状态。
- IPC 只传文件路径。
- IPC 不传视频二进制。
- IPC 不传大图 base64。
- 缩略图、预览视频、最终视频全部落盘。
- 前端用本地文件 URL 播放。

建议限制：

- 单条进度 JSON 小于 32KB。
- 缩略图尺寸控制在 320px 到 480px。
- 预览视频控制在几秒和低清。
- 日志只保留最近 N 行展示，完整日志写文件。

## 9. 硬件适配建议

### 低配机器

典型配置：

- 8GB 内存
- 4 核 CPU
- 无独显

建议：

- 默认稳定优先。
- chunk 60 到 90 秒。
- 预览 480p。
- 正式 1080p 但使用 veryfast。
- 避免高复杂度卡点策略做长片。

### 中配机器

典型配置：

- 16GB 内存
- 6 到 8 核 CPU
- 普通独显或核显

建议：

- chunk 120 秒。
- 预览 540p。
- 正式 1080p 30fps。
- 允许完整转场策略。

### 高配机器

典型配置：

- 32GB 以上内存
- 8 核以上 CPU
- NVIDIA / Apple Silicon

建议：

- chunk 180 秒。
- 可以启用更高质量 preset。
- 后续可接 NVENC / VideoToolbox / QSV。

## 10. GPU 加速建议

GPU 加速要谨慎做，不建议一开始强依赖。

优先级：

1. 先把 chunk 稳定和缓存做好。
2. 再做 FFmpeg 硬件编码探测。
3. 最后才做 UI 中的 GPU 开关。

Windows 可选：

- NVIDIA：`h264_nvenc`
- Intel：`h264_qsv`
- AMD：`h264_amf`

macOS 可选：

- `h264_videotoolbox`

注意：

- GPU 编码主要加速导出，不一定加速 MoviePy 的 Python 图像合成。
- 部分机器驱动不稳定，必须允许自动回退 `libx264`。

## 11. 推荐实施顺序

### 第一阶段：稳定止血

- 默认长片 chunk 120 秒。
- chunk cache key 完整化。
- 长视频强制稳定模式。
- 输出文件 atomic replace。
- 失败 chunk 可恢复。

### 第二阶段：速度提升

- 图片/背景/缩略图预处理缓存。
- 低清真实预览。
- FFmpeg 优先处理直切和简单淡化。
- 减少 MoviePy 最终全片 Composite。

### 第三阶段：产品化

- PyInstaller 打包 Python CLI。
- 本地 worker 服务化。
- 性能档位 UI。
- 硬件编码自动探测。
- 任务队列、暂停、取消、恢复。

## 12. 最终结论

要解决“短视频也慢、长视频内存疯涨、最终文件损坏”，不能只靠升级硬件，也不能继续把所有素材塞进一个 MoviePy timeline。

最成熟、可行、高效的策略是：

- 短视频走标准创作渲染。
- 长视频走 chunk 稳定渲染。
- 所有中间结果落盘缓存。
- 前端只传状态和路径。
- 预览低清化，正式生成分段化。
- FFmpeg 做它擅长的拼接和编码，MoviePy 只做复杂视觉片段。

这样才能同时兼顾效果、效率和普通机器的稳定性。
