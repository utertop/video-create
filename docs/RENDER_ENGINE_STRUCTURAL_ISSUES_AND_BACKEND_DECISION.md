## 目标

这份文档回答三个更尖锐的问题：

1. 现有底层渲染引擎的问题，到底是“普通工程优化还没做完”，还是“结构路线本身已经不对”。
2. 哪些问题可以继续在现有架构内解决，哪些问题已经接近现架构上限。
3. 到什么条件，才值得认真考虑引入新的长视频渲染后端，而不是继续在当前引擎上打补丁。

这份文档不是为了否定现有系统，而是为了避免两种极端：

- 一种是过早推翻重来，损失已经积累的 worker、cache、proxy、stable renderer、CI 和桌面端工程基础。
- 另一种是明明已经碰到结构性上限，却还在局部优化上持续消耗时间。

---

## 一、结论先行

当前渲染引擎的判断结论是：

- 它不是失败路线，仍然有明显优化价值。
- 它也不是成熟高性能后端，存在清晰的结构性短板。
- 当前最核心的问题不是“功能少”，而是“高性能主后端覆盖率不够高”。
- 现阶段还不应该全量推翻。
- 但后续是否替换或引入新后端，必须设立明确决策门槛，不能只靠感觉判断。

换句话说：

- 现在的问题，更像“现有引擎还停在可用但不够强的阶段”。
- 还没有到“必须整套重写”的程度。
- 但也已经不能把它当成天然成熟的底层渲染引擎来看。

---

## 二、现有引擎已经具备的价值

在讨论缺点之前，先明确它已经做对了什么。

当前系统已经具备这些重要基础：

- `scan -> plan -> compile -> render` 四阶段主链
- Python worker 协议与桌面端承载
- stable long-video renderer
- proxy media manifest
- visual base cache / segment cache / chunk cache
- `ffmpeg_direct_chunk`
- `ffmpeg_image_chunk`
- `ffmpeg_card_chunk`
- `ffmpeg_fitted_video_chunk`
- 保守型硬件编码默认策略
- build report / render scheduler / chunk scheduler
- packaged worker 与 CI smoke 守护

这说明它不是“只有想法，没有工程化”。  
它已经是一套成型的自动化视频生产内核，只是还没有把大多数常见场景都收进统一高性能后端。

所以后面所有判断，都应建立在这个前提上：

- 现有系统值得继续优化。
- 现有系统也必须接受更严格的结构性审视。

---

## 三、现有引擎的结构性问题清单

下面这些问题，不是单个 bug，也不是加几个参数就能消失的，它们属于结构层短板。

### 1. 后端混合得越来越多，但主后端还不够“主”

现在系统已经不是单一路径，而是混合后端：

- 一部分走 FFmpeg 直出
- 一部分走 FFmpeg fitted / motion fitted
- 一部分走 stable chunk concat
- 一部分走标准时间线
- 一部分仍然走 `MoviePy`

这本身不是坏事。成熟系统本来也会混合后端。  
真正的问题在于：

- 高性能路径还没有吃下足够大的场景比例
- `MoviePy` 仍然承担了不小一部分真实项目的重型渲染
- 很多性能结论仍然取决于项目是否恰好命中白名单

这意味着当前系统的高性能能力，仍然更像“局部快路径集合”，还不是“已经稳定吃下大多数常见场景的主后端”。

### 2. `MoviePy` 还没有彻底退到复杂场景兜底

如果只是作为兜底后端，`MoviePy` 没问题。  
真正的问题是它仍然会在下面这些场景里成为主成本来源：

- 复杂时间线合成
- 多层 overlay
- 某些图片动效
- 某些视频 motion
- 复杂转场
- 局部效果一旦超出白名单，就整体回到较重路径

所以当前的性能瓶颈，并不只是“编码慢”，更常见的是：

- 画面生成本身过重
- 画面生成的慢路径比例过高

这也是为什么只调编码器，往往不能解决 `10-15` 分钟项目 `3-4` 小时的问题。

### 3. 调度和缓存已经成型，但还没有完全进化成“可预测的渲染图”

现在系统已经有：

- route 判定
- stable chunk
- 局部失效
- visual base cache
- audio / image / card / video segment cache

这些都很对。  
但它离成熟渲染图系统还有差距：

- 某个片段为什么命中快路径，用户还不够容易理解
- 某个片段为什么失效，工程侧还不够容易解释
- 某些场景仍然是“碰到边界就整体放大失效范围”
- 路由决策、缓存决策、回退决策还没有完全统一成一套更稳定的中间层

也就是说，系统已经有“调度器雏形”，但还没有完全变成“成熟的渲染规划器”。

### 4. 引擎更偏自动化生产，不是成熟交互式 NLE 内核

这一点要明确区分。

现有引擎擅长的是：

- 自动扫描素材
- 自动生成结构
- 自动规划风格
- 自动批量导出

它不擅长的是：

- 高频细粒度人工拖拽编辑
- 多轨实时试听和即时修剪
- 强交互下的所见即所得时间线体验

所以如果目标是“像 Final Cut Pro 一样完整”，当前瓶颈并不只在渲染快不快，还在于底层数据模型还不是成熟 NLE 方向。

### 5. 成熟度短板比单点算法短板更大

当前问题不只是某个 motion 或转场实现不够优雅，而是：

- 路由覆盖率还不够高
- 回退边界还不够可解释
- 长项目在复杂特征叠加下仍可能掉回慢路径
- 预览、导出、缓存、代理、失败恢复的行为还没有达到“高度稳定可预期”

这类问题的本质是成熟度问题，不是单个特效问题。

---

## 四、哪些问题仍然适合继续在现架构内解决

下面这些问题，仍然明显适合继续在当前架构里做，不需要立刻换后端。

### 1. 快路径覆盖率不足

这是当前最值得继续优化的主问题。

包括：

- 扩更多安全视频段到 FFmpeg fitted / stable 路径
- 扩更多图片段、卡片段、轻 overlay 到快路径
- 继续收缩 `MoviePy` 参与面

这是现架构最应该继续挖的地方，而且已经证明有效。

### 2. stable renderer 进入条件和收益还可以继续做高

如果真实长视频还没有足够稳定地进入更适合的 stable 路线，那说明还有优化空间。

这类问题属于：

- 调度策略问题
- 阈值问题
- 路由策略问题

仍然属于现架构可继续推进的范围。

### 3. 编码、混音、缓存、代理的工程闭环还可以继续加强

包括：

- 更稳的硬件编码策略
- 更清晰的音频与画面解耦
- 更可控的缓存生命周期
- 更稳定的代理素材复用

这些都不是换后端的充分理由，因为它们首先是工程完整度问题。

### 4. 模板、配乐蓝图、自动策略层还远没有吃满

当前系统还有大量“策略层收益”可以继续做：

- 模板驱动默认 title / chapter / motion 风格
- 配乐蓝图对真实导出策略的细化映射
- 长视频性能模式的更保守策略

这些都能提升成片质量和自动结果稳定性，不需要先换后端。

---

## 五、哪些问题开始接近现架构上限

下面这些问题，如果以后反复出现，就说明已经越来越接近“继续补现引擎性价比下降”的边界。

### 1. 大量常见项目仍然长期落回 `MoviePy`

如果经过多轮快路径扩张后，真实项目里仍然经常出现：

- `moviepy_chunk` 很高
- `moviepy_required` 很高
- 路由大头仍然被复杂时间线吃掉

那就说明问题不再只是“白名单不够大”，而可能是：

- 当前主架构对复杂时间线的表达本身就天然偏重
- 想继续提速就必须考虑更适合时间线渲染的新后端

### 2. 继续扩白名单会明显带来视觉一致性风险

如果系统越来越依赖“只要证明安全就切 FFmpeg”，但每往前扩一步都要付出明显的：

- 字幕一致性风险
- 转场一致性风险
- 字体和布局兼容性风险
- 平台差异风险

那就说明现有“多条特殊快路径拼出来”的方式可能开始接近维护上限。

### 3. 多轨、复杂叠层、长时间线需求变成常态

如果产品未来明确要走向：

- 真正多轨
- 大量 layered overlay
- 更复杂字幕系统
- 局部调整层
- 实时时间线编辑

那现有自动化渲染内核虽然还可用，但不一定适合作为长期唯一主后端。

### 4. 性能瓶颈不再主要来自快路径缺失，而来自时间线模型本身

如果以后复盘发现：

- 不是哪个 segment 还没进快路径
- 也不是编码或缓存没命中
- 而是“项目一复杂，整体时间线结构就天然重”

这时就要承认：问题已经从“补优化”变成“底层时间线后端能力差距”。

---

## 六、什么时候还不该换后端

下面这些情况，不应作为换后端的理由。

### 1. 还没有跑过真实项目基线诊断

如果连真实 `build_report.json`、`route_counts`、编码命中情况都没有，就讨论换引擎，基本属于过早结论。

### 2. 当前主要问题仍然是快路径覆盖率不足

如果大头问题仍然是：

- 很多安全图片段没进 FFmpeg
- 很多安全视频段没进 stable
- 章节卡、轻字幕、轻 motion 还没吃满

那继续优化现架构通常更划算。

### 3. 当前主要问题仍然是工程不完整，而不是后端本体不行

比如：

- worker 分发还不稳
- cache 清理还不完善
- CI 守护还不够
- build report 还不够用

这些都不是换后端的理由。

### 4. 还没有给“长视频性能模式”一次完整机会

如果连更保守的长视频策略都还没系统化落地，就说现有引擎不行，结论通常太早。

---

## 七、什么时候应该认真考虑新后端

只有当下面这些条件同时或大部分成立时，才应该认真进入“评估新后端”阶段。

### 条件 A：现有低副作用优化路线基本做完

至少应包括：

- stable 进入策略已经较成熟
- FFmpeg 快路径已经覆盖大部分安全图片和安全视频场景
- 硬件编码策略已经稳定
- 代理、缓存、worker、CI 闭环已经成型

### 条件 B：真实项目仍然明显不达标

例如：

- `10-15` 分钟项目仍长期远高于预期
- 即使命中快路径后，整体导出仍无法接受
- 优化已经从“明显缩时”进入“边际收益很小”

### 条件 C：瓶颈已经坐实在复杂时间线能力，而不是局部快路径缺失

也就是数据上已经能说明：

- 大头不是编码
- 大头不是缓存缺失
- 大头不是简单 segment 还没进快路径
- 而是复杂时间线本身无法高效表达

### 条件 D：未来产品目标明确包含更强编辑器能力

如果未来目标明确要走向：

- 更完整的时间线交互
- 多轨
- 局部调整层
- 更丰富的可编辑特效体系

那就更有理由评估一个更适合时间线渲染的长期后端。

---

## 八、如果评估新后端，应该怎么评估

原则是不全量推翻，而是先把“新后端”当作 long-video stable backend 的候选实现。

### 1. 先替换长视频主路径，不替换整套架构

保留现有：

- `scan`
- `plan`
- `compile`
- worker 协议
- cache / report / proxy 基础

只把“复杂长视频正式导出后端”改成可插拔候选。

### 2. 先吃最常见、最安全的场景

候选新后端第一阶段只需要覆盖：

- 纯图片
- 纯视频
- 简单转场
- 简单字幕
- 基础多章节

复杂风格卡、特殊效果、复杂 overlay 继续保留旧路径兜底。

### 3. 必须要求可回退

任何新后端只要碰到不支持场景，必须自动回退，不影响最终可交付性。

### 4. 评估指标必须是工程指标，不是“理论更强”

真正要比的是：

- 真实项目耗时
- 画质一致性
- 字体与字幕一致性
- 失败率
- 回退能力
- 维护成本
- Windows 打包和分发成本

而不是只看“这个库看起来更专业”。

---

## 九、当前阶段的推荐判断

基于现阶段已知情况，推荐判断是：

### 1. 继续优化现有引擎，仍然是正确路线

尤其应继续做：

- 真实项目基线诊断
- 继续扩大安全视频段快路径
- 继续提高 stable 命中率
- 继续压缩 `MoviePy` 使用面

### 2. 现在就全量换后端，不是最佳时机

因为：

- 现有架构已经沉淀了很多可复用资产
- 现有快路径扩张仍然有明显收益空间
- 还没有充分证明“结构性上限已经真的到了”

### 3. 但必须开始用“决策门槛”看待后端替换问题

以后不应该再用模糊表述，比如：

- “感觉这套引擎不太行”
- “是不是直接换掉更快”

而应该按这份文档判断：

- 低副作用优化是否基本做完
- 真实项目是否仍然不达标
- 瓶颈是否已经坐实在复杂时间线能力
- 产品目标是否确实需要更强编辑器能力

只有这样，后端替换才会是理性决策，而不是情绪决策。

---

## 十、一句话结论

当前底层渲染引擎确实还不够优秀，但问题主要是：

- 高性能主后端覆盖率不足
- `MoviePy` 退场不够彻底
- 调度与缓存还没有完全成熟到“专业级可预测”

它还没有到必须推翻的阶段。  
现阶段正确策略是：

- 继续把现有系统打造成更成熟的混合渲染器
- 用真实项目数据验证它还能不能继续逼近目标
- 只有当低副作用优化路线基本走完后仍明显不达标，才认真推进新后端评估
