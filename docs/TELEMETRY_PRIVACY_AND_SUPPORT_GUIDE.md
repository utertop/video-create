# 匿名遥测、隐私与支持指南

## 目的

这套能力用于帮助团队回答以下产品成熟度问题：

- 当前版本的 `crash-free session` 是多少
- 首次导出成功率是否达到目标
- 最常见失败码、支持队列、恢复标签分别是什么
- 稳定渲染的 `resume / retry / partial reuse` 是否真的在帮助用户恢复任务

它的设计目标不是“收集更多用户数据”，而是让稳定性、支持分流和版本判断有真实依据。

当前正式同意文本版本见：

- [TELEMETRY_PRIVACY_NOTICE_V2026_05.md](./TELEMETRY_PRIVACY_NOTICE_V2026_05.md)

## 当前实现边界

当前版本默认关闭，用户需要显式开启“匿名遥测（可选）”。

开启后，仅记录匿名事件标签和聚合计数，例如：

- session 是否正常结束
- 首次导出是否成功
- render 成功 / 失败
- error code
- support queue
- severity
- tags
- recovery 是否可恢复 / 可重试

## 不记录的内容

当前实现不记录以下内容：

- 素材文件内容
- 视频内容或缩略图
- 原始媒体路径
- 项目标题、片尾文本、水印文案
- 用户自定义脚本或命令输入

诊断包里仍然可能包含项目路径、输出路径和日志，这属于“支持诊断信息”，不是匿名遥测本身。

## 数据位置

遥测历史保存在应用本地数据目录中的 `telemetry_store.json`。

这意味着：

- 当前能力是本地聚合，不依赖远程服务
- 用户可以随时关闭
- 用户可以随时清空本地历史

## 用户治理能力

界面中已经提供：

- 开启 / 关闭匿名遥测
- 查看本地稳定性摘要
- 清空本地匿名遥测历史
- 通过诊断包导出当前遥测摘要

清空历史后，当前会话会从新的统计基线继续，不会影响项目文件。

## 支持使用建议

支持同学处理问题时，建议优先结合以下三类信息：

1. `errorCodeReport`
2. `supportCaseReport`
3. `telemetryReport`

推荐判断顺序：

1. 先看本次故障的错误码和建议动作
2. 再看是否属于 `render-recovery / environment / project-validation / app-runtime`
3. 最后看本地长期指标是否提示某个失败码或支持队列持续升高

## 下一步建议

如果后续进入“真正的 crash reporting / 可选远程遥测”阶段，建议保持以下原则：

- 继续默认关闭，显式同意后才上传
- 远程上传只发送匿名标签、版本号和聚合字段
- 诊断包与远程事件使用同一套错误码 / support queue / recovery 标签
- 为每个版本保留清晰的“收集什么、不收集什么、如何关闭、如何清空”说明
